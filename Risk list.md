first list:
    1. The device has zero way to recover from a mistake mid-typing, beyond "double-tap to wipe everything." Your doc says double-tap on an empty buffer exits to the main menu, and back undoes a character or sentence. For a sighted person, "undo" is visual — you see what got removed. For a blind user, every undo needs to be spoken immediately and exactly ("removed: dot. Now reads: 'hello worl'") or they will not trust the undo and will panic-delete the whole note instead. This is a user-experience failure waiting to happen, not a wiring one.
    2. Audio feedback timing vs. typing speed. A fast Braille typist commits a character every second or two. If a confirmation sound or speech plays for every character and that audio takes longer than the time before the next press, audio will start stacking, overlapping, or cutting itself off mid-word. This is extremely common in talking devices and is rarely caught until a real fast user tests it. You should specifically test: "what happens if I type the next letter before the device finishes saying the last one?"
    3. No physical orientation cue. Once it's a sealed, modular box, how does the user find Dot 1 versus Dot 6 by touch alone, in the dark, without looking? Real Braille devices solve this with a raised dot or asymmetric button shape near one specific key. This isn't in your doc at all yet, and it's the kind of thing that makes a device usable versus frustrating in real life.
    4. Battery / power-loss note safety. Nowhere in the doc does it mention what happens to /draft.txt if power is cut mid-sentence. For a note-taking device used daily, an unexpected shutdown losing the in-progress note is the single most trust-destroying failure a user can hit. This needs an explicit test: type half a note, pull power, power back on — does the draft come back?
    5. The "storage full" and "AI failed" spoken messages need to be genuinely actionable, not just informative. Telling a blind user "storage full" doesn't tell them what to do next. Does the device guide them to delete a note right then, or do they need to navigate there themselves with no further help? This is worth deciding deliberately rather than letting it just be whatever the current message defaults to.
